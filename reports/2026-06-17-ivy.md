> 今日重点：点赞全链路、腾讯天御审核（文本/图片分离）、附近信息流落地、高德地图与位置分享

## 一、点赞互动（前后端打通）

把「动态」补上了最基础的互动能力——点赞。

**后端**
- Store 接口与 PostgresStore 实现 LikePost / UnlikePost，支持用户对帖子点赞与取消
- Post 模型新增点赞数（like count），点赞操作保证幂等
- FeedCursor 模型支持基于点赞的「热门」信息流游标
- 新增 like / unlike 路由，返回更新后的点赞数
- 补充点赞功能的测试，覆盖幂等与既有功能集成

**移动端**
- DynamicContent 新增 likeCount、likedByMe 字段跟踪点赞状态
- 详情页与信息流卡片都加上点赞按钮，支持点赞/取消
- 乐观更新（optimistic UI）：FeedStore 本地维护点赞状态，无需额外网络往返
- 新增 PostStatus 枚举管理帖子可见状态

## 二、内容审核（接入腾讯天御）

为帖子发布加上审核闭环，避免违规内容直接公开。

**后端**
- 新增 moderation 模块：定义审核服务接口，实现腾讯「天御」审核 API 接入
- 审核服务支持 mock 与真实服务按环境解析（resolve）切换
- 帖子创建初始状态改为 pending，发布前先过审核检查
- 自动审核 + 人工审核双流程；新增 admin 审核队列路由（通过/驳回）
- 数据库模型与迁移新增审核记录表，保留审核决策审计轨迹
- 帖子作者可查看自己尚未发布的帖子，同时维持对外可见性规则

**fsp-admin（运营后台）**
- 新增 ModerationQueue 审核队列页面，管理待审核帖子
- API client 增加审核队列能力：拉取待审、通过、驳回
- App 路由 + AdminLayout 导航补充审核队列入口
- 审核项表格视图，带通过/驳回操作

## 三、信息流分发频道（设计落地）

为信息流规划三个分发频道，各自独立的数据基础与排序策略。

- 三个频道：**好友**（社交图谱）、**附近**(地理位置)、**热门**(基于点赞的热度信号)
- 分阶段推进：每个频道在自己的数据底座上独立运行
- 每个频道只展示相关帖子，未实现的频道给出明确空状态
- 同步补充 openspec：feed-distribution-channels、post-editing、post-moderation 三个 change 的 proposal / spec / tasks

## 四、审核策略细化（天御 TMS/IMS 分离）

在凌晨接入的基础上，下午把审核策略做了拆分细化。

**后端**
- 重构腾讯天御审核服务，文本（TMS）与图片（IMS）走各自独立的审核策略
- 文本、图片分别按对应能力调用，结果分流处理

**移动端**
- 梳理并明确天御审核策略，更新相关文档

## 五、附近信息流与高德地图

把「附近」频道从设计推进到可用，并接入真实地图。

**后端**
- 增强「附近」信息流，支持基于地理位置（geolocation）的内容拉取

**移动端**
- Android 端接入高德（AMap）真实地图
- DynamicPostScreen 与 FeedStore 新增位置分享功能（发帖带定位、信息流展示）

---

**总结**：今天围绕「动态」的互动与治理补齐了多块——点赞做了前后端全链路（含乐观更新与幂等）；内容审核接入腾讯天御并搭起 pending → 自动/人工审核 → 发布的闭环，下午进一步把文本（TMS）/图片（IMS）拆成独立策略；信息流的「附近」频道从设计落地为基于地理位置的实际拉取，移动端 Android 接入高德真实地图并支持发帖位置分享；好友/附近/热门三频道设计也写进了 openspec。
